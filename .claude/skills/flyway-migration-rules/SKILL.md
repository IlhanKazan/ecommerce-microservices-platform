# Flyway Migration Kuralları

Bu skill DB schema değişikliği yaparken uyulması gereken kuralları içerir. **CONVENTIONS.md §4.6** + **CLAUDE.md "Outbox schema değişikliği runbook"**'un pratik genişletmesi.

## Değişmez kural 1: Mevcut V dosyalarına DOKUNMA

`src/main/resources/db/migration/V1__init.sql`, `V2__xxx.sql` gibi **zaten var olan dosyalar değiştirilmez**. Schema değişikliği için **yeni** `V(N+1)__aciklama.sql` yazılır.

Sebep: Flyway her migration'ın **checksum**'unu DB'de tutar. Dosyayı değiştirirsen checksum mismatch, servis boot edemez.

**Dev ortamında bile dokunma.** Dev'de checksum patladığında tek çare volume silip sıfırdan başlatmak:
```bash
docker compose down -v
rm -rf infrastructure/postgres_data  # proje root'undan
docker compose up -d
```

Volume silmek yerine mevcut V'yi düzenlemek "shortcut" gibi görünür ama zincirleme bug üretir — takımda farklı checksum'lar, test environment'larda tutarsızlık.

## Değişmez kural 2: ddl-auto: validate

Tüm servisler `spring.jpa.hibernate.ddl-auto: validate` kullanır. Hibernate DB'yi değiştirmez, sadece entity ↔ tablo eşleşmesini kontrol eder.

Sonuç: **Her schema değişikliği mutlaka bir V dosyasından gelir.** Entity'ye kolon eklersen ve V dosyası yazmazsan, servis boot'ta `SchemaManagementException` atar.

Servis build sırasında test profil'i `create-drop` kullanabilir (`application-test.yml`). O ayrı.

## V dosyası isim formatı

```
V{sayı}__{kısa_aciklama}.sql
```

- **Çift underscore** `__` — tek olursa Flyway tanımaz
- Sayı int, artan; sıçrama olabilir ama aynı sayı iki kez olmaz
- Açıklama snake_case

Örnekler:
- `V1__init_stock.sql`
- `V2__rename_outbox_column.sql`
- `V3__migrate_outbox_id_and_enrich_inbox.sql`
- `V4__outbox_replica_identity_full.sql`
- `V5__add_inbox_indexes.sql`

## Değişiklik tipine göre şablonlar

### A) Kolon ekleme (backward-compatible)

```sql
ALTER TABLE products
    ADD COLUMN IF NOT EXISTS seo_title VARCHAR(255),
    ADD COLUMN IF NOT EXISTS seo_description TEXT;
```

`IF NOT EXISTS` idempotent olması için.

### B) Kolon silme (breaking — entity'den de sil)

```sql
ALTER TABLE products
    DROP COLUMN IF EXISTS old_field;
```

Önce entity'den sil, mvn build'le test et (validate geçer mi), sonra V yaz. Sırayı karıştırma.

### C) Kolon rename

```sql
ALTER TABLE outbox RENAME COLUMN event_type TO message_type;
```

Rename ile çoğu zaman Debezium connector config'de de değişiklik gerekir (bu bizim V2'de yaşandı). `debezium-connector-config` skill'ine bak.

### D) Kolon tipi değişikliği

Basit tip değişimi (VARCHAR uzunluğu vb.):
```sql
ALTER TABLE x ALTER COLUMN name TYPE VARCHAR(500);
```

Karmaşık (UUID → BIGINT gibi) cast yapılamayacak dönüşümler: **drop+recreate** önerilir. Dev ortamında veri önemsiz, bu en temiz yol.

### E) Drop+recreate (karmaşık migration)

```sql
-- Önce mevcut veri backup lazım mı? Dev'de genelde hayır.
DROP TABLE IF EXISTS outbox;

CREATE TABLE outbox (
    id              BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    aggregate_type  VARCHAR(100) NOT NULL,
    aggregate_id    VARCHAR(255) NOT NULL,
    message_type    VARCHAR(255) NOT NULL,
    message_payload JSONB NOT NULL,
    created_at      TIMESTAMP WITHOUT TIME ZONE NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_outbox_created_at ON outbox(created_at);

-- Debezium için kritik — unutma
ALTER TABLE outbox REPLICA IDENTITY FULL;
```

**REPLICA IDENTITY FULL drop+recreate sonrası mutlaka tekrar set edilmeli** — tablonun default'u `DEFAULT` (primary key only).

### F) Zenginleştirme migration'ı (kolon ekleme + veri doldurma)

```sql
-- 1. Yeni kolonları ekle (nullable ki doldurabilesin)
ALTER TABLE inbox
    ADD COLUMN status        VARCHAR(50),
    ADD COLUMN retry_count   INTEGER DEFAULT 0,
    ADD COLUMN error_message TEXT,
    ADD COLUMN received_at   TIMESTAMP WITHOUT TIME ZONE;

-- 2. Mevcut satırları doldur
UPDATE inbox
SET status      = 'PROCESSED',
    retry_count = 0,
    received_at = COALESCE(processed_at, NOW())
WHERE status IS NULL;

-- 3. NOT NULL kısıtı ekle (veri doldurulduktan sonra)
ALTER TABLE inbox
    ALTER COLUMN status      SET NOT NULL,
    ALTER COLUMN received_at SET NOT NULL;

-- 4. processed_at artık nullable (PENDING kayıtlarda null)
ALTER TABLE inbox
    ALTER COLUMN processed_at DROP NOT NULL,
    ALTER COLUMN processed_at DROP DEFAULT;

-- 5. Index'ler
CREATE INDEX IF NOT EXISTS idx_inbox_status ON inbox(status);
CREATE INDEX IF NOT EXISTS idx_inbox_pending ON inbox(received_at) WHERE status = 'PENDING';
```

Sıra kritik: add → fill → NOT NULL. Ters sırada yaparsan var olan satırlardan dolayı NOT NULL set edilemez.

## Outbox tablosunu etkileyen migration → Debezium çakışması

Bu en sık görülen operasyon. `outbox` tablosuna dokunan migration yazdığında Debezium replication slot'u bozulabilir.

**Risk düzeyi:**
- Kolon ekleme/NOT NULL toggle → genelde risksiz
- Rename → connector config de güncellenmeli ama slot sağlam kalır
- Tip değişikliği (UUID→BIGINT) veya drop+recreate → slot invalidate olur

**Risk varsa:** `debezium-connector-config` skill'indeki runbook'u uygula. Özet:

```bash
# Servisi durdur
docker stop <service>

# Connector'ı sil
curl -X DELETE http://localhost:8083/connectors/<service>-connector

# Slot'u drop et
docker exec -it postgres psql -U $POSTGRES_USER -d <service>_db \
  -c "SELECT pg_drop_replication_slot('<service>_debezium_slot');"

# Servisi başlat (Flyway V çalışsın)
docker start <service>
docker logs <service> 2>&1 | grep -iE 'migrat|flyway' | head -20

# Connector'ları re-register et
cd infrastructure/scripts && ./register_connector.sh
```

## "Önce mi sonra mı?" checklist

Schema değiştirip entity'yi de güncelleyeceğin zaman doğru sıra:

1. **Önce entity'yi düzenle** — field ekle/sil/rename
2. **Sonra V dosyasını yaz** — entity'nin yeni haline uygun
3. **Build al** — `./mvnw compile -DskipTests` (Hibernate validate hata vermemeli eğer V doğruysa)
4. **Servisi restart** — Flyway migration uygulanır, validate OK
5. **Outbox'a dokunduysan** — Debezium slot cleanup runbook'unu çalıştır

Ters sırada yaparsan:
- Önce V yazarsan, entity eski halde → validate patlar, servis bootlamaz
- Build almadan restart edersen → sürpriz hatalar

## Dev-only kısayol: tam reset

Çok şey ters gitti, migration zincirinde karmaşıklık var:

```bash
# 1. Her şeyi durdur
docker compose down -v

# 2. Volume'leri temizle
rm -rf infrastructure/postgres_data
rm -rf infrastructure/zookeeper_data
rm -rf infrastructure/kafka_data

# 3. Altyapıyı baştan ayağa kaldır
docker compose up -d postgres kafka debezium zookeeper keycloak minio

# 4. Debezium hazır olunca register script
./infrastructure/scripts/register_connector.sh

# 5. Servisleri başlat (Flyway sıfırdan tüm V'leri uygular)
```

Prod'da olsa bu yıkıcı, ama dev'de serbest. TÜBİTAK ve staj kapsamı, veri koruma derdi yok.

## Referans migration örnekleri

Mevcut projede:

- **`backend/product-service/src/main/resources/db/migration/V2__outbox_CDC_config.sql`** — `REPLICA IDENTITY FULL` ekleme örneği
- **`backend/stock-service/.../V3__migrate_outbox_id_and_enrich_inbox.sql`** — drop+recreate + zenginleştirme + REPLICA IDENTITY kombinasyonu
- **`backend/stock-service/.../V4__outbox_replica_identity_full.sql`** — drop+recreate sonrası unutulmuş REPLICA IDENTITY'i düzelten sonraki V
- **`backend/stock-service/.../V5__add_inbox_indexes.sql`** — basit index ekleme
- **`backend/payment-service/.../V3__migrate_outbox_inbox_for_base_classes.sql`** — kolon drop + REPLICA IDENTITY
- **`backend/user-tenant-service/.../V4__...`** — aynı pattern

Yeni migration yazmadan önce **mutlaka** en yakın referansa bak, stili ve yorum satırlarını kopyala.

## Tuzaklar

### 1. Çift underscore unutma

`V3_migrate_outbox.sql` → Flyway tanımaz. `V3__migrate_outbox.sql` doğru.

### 2. Mevcut V'yi "küçük bir düzeltme" için değiştirme

Dev'de "nasılsa reset atarım" diye değiştirmek cazip, yapma. Alışkanlık olunca prod'da yakacak.

### 3. drop+recreate sonrası REPLICA IDENTITY

Yeni tabloda REPLICA IDENTITY default'a döner. Debezium outbox tablosundaki update/delete event'lerini kaçırır. Migration'ın sonuna `ALTER TABLE outbox REPLICA IDENTITY FULL;` ekle.

### 4. NOT NULL sırası

Zenginleştirme migration'ında: add nullable → UPDATE → SET NOT NULL. Ters sırada var olan satırlardan dolayı NOT NULL set edilemez.

### 5. CONCURRENTLY ile index

Büyük tablolarda `CREATE INDEX CONCURRENTLY` Postgres'in önerisi ama Flyway transaction'ında çalışmaz. Flyway'e `-- flyway:ignoreMissing` yerine migration'ı iki V'ye böl veya `CREATE INDEX IF NOT EXISTS` kullan (dev'de problem değil).
